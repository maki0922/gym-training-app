# ジム顧客トレーニング管理Webアプリ — 要件定義書

## 1. プロジェクト概要

小規模パーソナルジム向けのトレーニング記録・管理Webアプリ。
トレーナーが顧客のトレーニング情報を記録・閲覧し、当日の種目選定に活用する。

**アプリ名**: GymLog（仮称。ロゴ・正式名称は後日決定。変更しやすいよう、アプリ名は定数として一箇所で管理すること）

### 想定ユーザー

- **オーナー（複数名可）**: 全管理機能を持つ（トレーナーの招待・削除、種目マスタの編集・無効化、顧客の無効化など）。オーナー兼トレーナーとしてトレーニング記録も入力可能。最初に登録したオーナー（プライマリオーナー）は削除不可
- **トレーナー（10名未満）**: トレーニング記録の入力・閲覧、顧客の追加・編集、種目マスタの追加
- **顧客**: ファーストリリースではログイン不可。将来的に自分の記録を閲覧できるようにする想定

### 利用デバイス

- トレーニング中の記録入力: **スマホ**（モバイルファースト）
- ユーザー管理・種目マスタ管理: **PC**（デスクトップ）
- → レスポンシブデザイン必須

## 2. 技術スタック

### 必須要件

| 区分 | 技術 | 備考 |
|------|------|------|
| フロントエンド | **Next.js (App Router)** + **TypeScript** | Vercelとの親和性、型安全性 |
| UIライブラリ | **Tailwind CSS** + **shadcn/ui** | シンプル・清潔感のあるデザイン |
| バックエンド / DB | **Supabase** (PostgreSQL) | Auth, Database, RLS |
| 認証 | **Supabase Auth** (メール + パスワード) | |
| デプロイ | **Vercel** | |
| 言語 | **日本語** UI | 英語対応は将来検討 |

### セキュリティ要件

- Supabase **Row Level Security (RLS)** を全テーブルに適用すること
- API経由のデータアクセスは認証済みユーザーのみ許可
- トレーナーは自ジムのデータのみアクセス可能
- オーナーのみがトレーナーの招待・削除を実行可能
- 環境変数で機密情報を管理し、クライアントに露出させない（必要な環境変数: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SITE_URL`）
- Server Actionsまたは Route Handlers を活用し、Supabase service_role keyをサーバーサイドのみで使用

## 3. データモデル

### テーブル設計

```
profiles
├── id: uuid (PK, auth.usersと紐付け)
├── email: text
├── display_name: text
├── role: enum ('owner' | 'trainer')
├── is_primary: boolean (default: false, 最初に登録したオーナーのみtrue。削除不可)
├── created_at: timestamptz
└── updated_at: timestamptz

customers (顧客)
├── id: uuid (PK)
├── name: text (氏名)
├── name_kana: text (フリガナ)
├── gender: enum ('male' | 'female' | 'other') [任意]
├── date_of_birth: date [任意]
├── notes: text [任意、メモ]
├── is_active: boolean (default: true)
├── created_at: timestamptz
└── updated_at: timestamptz

exercise_master (種目マスタ)
├── id: uuid (PK)
├── name: text (種目名)
├── category: text (部位カテゴリ: 胸, 背中, 脚, 肩, 腕, 腹, 有酸素, その他)
├── is_system: boolean (default: false, 初期登録種目かどうか)
├── is_active: boolean (default: true)
├── created_at: timestamptz
└── updated_at: timestamptz

training_sessions (トレーニングセッション)
├── id: uuid (PK)
├── customer_id: uuid (FK → customers)
├── trainer_id: uuid (FK → profiles)
├── session_date: date
├── status: enum ('in_progress' | 'completed') (default: 'in_progress')
├── notes: text [任意、セッション全体メモ]
├── is_deleted: boolean (default: false, 論理削除フラグ)
├── created_at: timestamptz
└── updated_at: timestamptz

training_records (トレーニング記録 — セッション内の各種目)
├── id: uuid (PK)
├── session_id: uuid (FK → training_sessions)
├── exercise_id: uuid (FK → exercise_master, NULL許可)
├── exercise_name_manual: text [exercise_idがNULLの場合に自由入力名]
├── sort_order: integer (表示順)
├── notes: text [任意、種目ごとのメモ]
├── created_at: timestamptz
└── updated_at: timestamptz

training_sets (セット記録)
├── id: uuid (PK)
├── record_id: uuid (FK → training_records)
├── set_number: integer
├── weight: decimal [kg]
├── reps: integer [回数]
├── notes: text [任意、セットごとのメモ。例: 「補助あり」「フォーム崩れ」]
├── created_at: timestamptz
└── updated_at: timestamptz
```

### RLSポリシー方針

- 全テーブル: 認証済みユーザー（トレーナー・オーナー）のみアクセス可能
- profiles: オーナーのみ他ユーザーの招待・削除が可能。`is_primary: true` のオーナーは削除不可
- exercise_master: 全ロールが閲覧・追加可能。編集・無効化はオーナーのみ
- customers: 全ロールが閲覧・追加・編集可能。無効化（is_active → false）はオーナーのみ
- training系テーブル: 全トレーナー・オーナーが読み書き可能（同ジム内共有）

### 権限一覧

| 機能 | オーナー | トレーナー |
|------|:--------:|:----------:|
| トレーナーの招待・削除 | ○ | × |
| 種目マスタの追加 | ○ | ○ |
| 種目マスタの編集・無効化 | ○ | × |
| 顧客の追加・編集 | ○ | ○ |
| 顧客の無効化（退会処理） | ○ | × |
| トレーニング記録の入力・閲覧・編集 | ○ | ○ |
| セッション内の種目・セットの削除 | ○ | ○ |
| セッション自体の削除 | ○ | × |

### データ削除ルール

- **セッション内の種目・セットの削除**: 全ロールが可能。入力中に種目変更やセット取り消しができるようにする。削除前に確認ダイアログを表示する
- **セッション自体の削除**: オーナー権限を持つユーザーのみ可能。トレーナーにはセッション削除ボタンを表示しない。削除は論理削除とし、DBからは物理削除しない
- **種目マスタの削除**: 物理削除は行わない。無効化（is_active → false）で対応し、過去の記録との整合性を保つ

### トレーナー削除時のデータ扱い

- トレーナーの削除は論理削除（アカウント無効化）とし、過去のセッション記録はすべて残す
- 該当トレーナーが担当したセッションの閲覧は引き続き可能
- セッション詳細画面では「担当: ○○（退職済み）」のように表示する

## 4. 画面一覧と機能

### 4.1 認証

| 画面 | 機能 |
|------|------|
| ログイン画面 | メール + パスワードでログイン |
| パスワードリセット | メールによるリセットフロー |

※ 新規アカウント作成はオーナーからの招待のみ（セルフ登録不可）

### 4.2 トレーナー向け画面（モバイルファースト）

| 画面 | 機能 |
|------|------|
| **ダッシュボード** | 本日のセッション一覧、直近の記録へのクイックアクセス |
| **顧客一覧** | 顧客の検索・選択 |
| **顧客詳細** | 顧客の基本情報、過去セッション一覧（日付降順） |
| **セッション詳細（閲覧）** | 過去のセッションの記録を閲覧（種目・重量・回数・セット・メモ） |
| **セッション記録（入力）** | 新規セッションの作成。種目をマスタから選択 or 自由入力し、セットごとに重量・回数を入力。セッション全体メモ・種目ごとのメモも入力可能 |
| **セッション編集** | 既存セッションの編集 |

#### セッション記録画面の操作フロー（重要）

**データ保存方式: 自動保存（リアルタイム保存）**
- 「新規セッション開始」時点でセッションをDBに作成（status: `in_progress`）
- 種目追加・セット追加・メモ入力など、各操作のたびにDBへ即時保存する
- ブラウザの「戻る」やアプリ終了をしてもデータは消えない
- 最後に「セッション完了」ボタンを押すと status が `completed` に変わり、正式な記録として確定される

**入力途中セッションの扱い**
- `in_progress` のセッションがある場合、ダッシュボードや顧客詳細画面に「入力途中のセッションがあります」と通知を表示し、タップで入力を再開できる
- 他のトレーナーが顧客の過去記録を閲覧する際、`in_progress` のセッションは表示しない（または「入力中」ラベルを付けて区別する）
- **Vercel Cron Job**: 毎日0:00（JST）に実行し、session_dateが前日以前かつ `in_progress` のセッションを自動的に `completed` に変更する。押し忘れによるデータ放置を防止する

**操作ステップ**

1. 顧客を選択
2. 「新規セッション開始」ボタン
   - セッションがDBに作成される（status: `in_progress`）
   - **重複防止**: 同じ顧客に対して同日に既に `in_progress` または `completed` のセッションが存在する場合、警告ダイアログを表示する
     - 表示内容: 「本日のセッションが既に存在します（担当: ○○トレーナー / ステータス: 入力中 or 完了）」
     - 選択肢: 「既存のセッションを開く」または「それでも新規作成する」（1日2回トレーニングのケースに対応）
3. 「種目追加」→ 以下の2段階で選択（種目数が100以上あるため、一覧から直接選ばせない）
   - **ステップ1**: 部位カテゴリを選択（胸, 背中, 脚, 肩, 腕, 腹, 有酸素, その他）
   - **ステップ2**: 選択した部位に属する種目一覧から選択
   - **自由入力**: マスタにない種目はテキストで自由入力も可能（部位選択画面に「自由入力」ボタンを設置）
   - → 種目追加と同時にDBに保存される
4. 各種目ごとにセットを追加（重量 kg / 回数 / セットごとのメモ）→ セット追加・変更のたびにDBに保存される
5. 前回同種目の記録があれば参考値として表示（前回の重量・回数・セット数）
   - **「前回をコピー」ボタン**: タップすると前回のセット数・重量・回数をそのまま入力フィールドにコピーする。トレーナーは変更したい箇所だけ修正すればよい
6. セットごとのメモを任意入力（例: 「補助あり」「フォーム崩れ」など）
7. 種目ごとのメモを任意入力
8. セッション全体のメモを任意入力
9. 「セッション完了」ボタン → status が `completed` に変更され、正式な記録として確定

### 4.3 オーナー向け管理画面（PC向け）

| 画面 | 機能 |
|------|------|
| **トレーナー管理** | トレーナー・オーナーの一覧表示、招待（メールアドレス入力 + ロール選択 → 招待メール送信）、削除（論理削除）。プライマリオーナーは削除ボタンを非表示 |
| **種目マスタ管理** | 種目の一覧・追加・編集・無効化。カテゴリ別フィルタリング。※トレーナーは追加のみ可能（トレーナー向け画面からも種目追加できる） |
| **顧客管理** | 顧客の追加・編集・無効化。※トレーナーは追加・編集のみ可能、無効化ボタンはオーナーにのみ表示 |

## 5. 非機能要件

- **パフォーマンス**: 記録入力画面はスマホで快適に操作できること（タップ数を最小限に）
- **データバックアップ**: Supabase標準のバックアップに依存
- **ログ**: 重要な操作（セッション作成・削除、トレーナー招待・削除）のログを残す
- **将来拡張**: 顧客ログイン機能を追加しやすいデータ設計にしておくこと

## 6. UI/UXガイドライン

- **デザインテイスト**: シンプル・清潔感・ミニマル。白を基調に、アクセントカラーは控えめに
- **フォント**: Noto Sans JP または同等の日本語Webフォント
- **モバイル操作**: ボタンは十分な大きさ（44px以上のタップターゲット）、入力フィールドは数値キーボードが表示されるよう `inputmode="numeric"` を使用
- **ナビゲーション**: モバイルはボトムナビゲーション、PCはサイドバー
- **ローディング**: スケルトンUIまたはスピナーで読み込み中を表示

## 7. 初期データ（シード）

### 種目マスタ初期データ（is_system: true）

以下のカテゴリと代表的な種目をシードデータとして投入する:

- **胸**: ベンチプレス, ダンベルフライ, インクラインベンチプレス, チェストプレス（マシン）, ケーブルクロスオーバー
- **背中**: デッドリフト, ラットプルダウン, シーテッドロウ, ベントオーバーロウ, 懸垂
- **脚**: スクワット, レッグプレス, レッグカール, レッグエクステンション, ブルガリアンスクワット
- **肩**: ショルダープレス, サイドレイズ, フロントレイズ, リアレイズ, アーノルドプレス
- **腕**: アームカール, トライセプスプレスダウン, ハンマーカール, フレンチプレス
- **腹**: クランチ, プランク, レッグレイズ, アブローラー
- **有酸素**: トレッドミル, エアロバイク, ローイングマシン
- **その他**: ストレッチ, フォームローラー

## 10. 補足・注意事項

- **Supabase CLIでマイグレーション管理**を行うこと（`supabase migration new` / `supabase db push`）
- **TypeScriptの型はSupabase CLI**で自動生成すること（`supabase gen types typescript`）
- **エラーハンドリング**: ユーザー向けに日本語のエラーメッセージを表示
- **バリデーション**: zodを使用してフォーム入力を検証
- **状態管理**: React Server Componentsを最大限活用し、クライアント状態は最小限に
- **コミットメッセージ**: Conventional Commits形式
