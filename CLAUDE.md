# CLAUDE.md

## プロジェクト概要

小規模パーソナルジム向けのトレーニング記録・管理Webアプリ「GymLog（仮称）」

## 要件定義書

**必ず最初に読むこと**: `docs/requirements.md`

要件定義書にはデータモデル、画面一覧、操作フロー、権限設計、データ保護方針、ディレクトリ構成がすべて記載されている。開発はこのドキュメントに従って進めること。

## 技術スタック

- **フロントエンド**: Next.js (App Router) + TypeScript
- **UI**: Tailwind CSS + shadcn/ui
- **バックエンド / DB**: Supabase (PostgreSQL, Auth, RLS)
- **デプロイ**: Vercel
- **言語**: 日本語 UI

## 開発ルール

### コード規約

- 言語: TypeScript（strict mode）
- フォーマッタ: Prettier
- リンター: ESLint（Next.js推奨設定）
- コミットメッセージ: Conventional Commits 形式（例: `feat: 顧客一覧画面を追加`、`fix: セッション保存時のバリデーションエラーを修正`）

### Supabase

- マイグレーション管理: `supabase migration new <name>` → `supabase db push`
- 型の自動生成: `supabase gen types typescript --project-id <project-id> > src/lib/types/database.types.ts`
- **全テーブルにRLSを適用すること**（RLSなしのテーブルは許可しない）
- service_role key はサーバーサイド（Server Actions / Route Handlers）のみで使用。クライアントには絶対に露出させない

### フォーム・バリデーション

- zodでスキーマ定義 → サーバーサイドとクライアントサイドの両方でバリデーション
- エラーメッセージは日本語で表示

### 状態管理

- React Server Components を最大限活用
- クライアント状態は最小限に（必要な場合のみ `useState` / `useReducer`）
- データフェッチは Server Components または Server Actions で行う

### UI

- shadcn/ui コンポーネントを優先して使用
- モバイルファースト（スマホでの記録入力が最も重要なユースケース）
- タップターゲット: 44px 以上
- 数値入力フィールド: `inputmode="numeric"` を使用

### セキュリティ

- 環境変数は `.env.local` で管理（`.gitignore` に含めること）
- 権限チェックはサーバーサイドで必ず行う（クライアント側の表示制御だけに頼らない）
- API Route / Server Actions でのユーザー認証チェックを必須とする

## 開発の進め方

`docs/requirements.md` のセクション8「開発の進め方」に記載のフェーズ1ステップに従い、1サブステップずつ進めること。

- ステップは1〜13、各ステップ内にサブステップ（例: 1-1, 1-2, ...）がある
- 各ステップ完了時にコミットする
- 次のステップに進む前に動作確認を行う
- 問題が発生した場合は、そのステップ内で解決してから先に進む

### 機能仕様書ルール

ステップの種類に応じて、仕様書の要否が異なる。

**仕様書不要（セットアップ系）**: ステップ1〜4
- プロジェクト初期設定、DB設計、認証、レイアウトなどは要件定義書の記載で十分
- そのまま実装に進んでよい

**仕様書必須（機能系）**: ステップ5〜11
- 各ステップの着手前に、機能仕様書を `docs/specs/` に作成すること
- ファイル名: `step{N}-{機能名}.md`（例: `step5-trainer-management.md`）
- 仕様書に含めるべき内容:
  - 画面ごとの表示項目・レイアウト
  - ユーザー操作ごとの振る舞い（タップ/クリック → 何が起きるか）
  - 入力値のバリデーションルール（型、上限・下限、必須/任意、文字数制限）
  - エラー時の表示・挙動
  - 権限による表示・操作の違い
- 仕様書は開発者（人間）がレビュー・承認してから実装に進むこと

**仕様書不要（仕上げ系）**: ステップ12〜13
- レスポンシブ調整、最終確認、デプロイは仕様書不要

## 重要な設計判断（要件定義で決定済み）

- **自動保存方式**: セッション記録は操作のたびにDBに即時保存する。「保存」ボタンではなく「セッション完了」ボタンでステータスを確定する
- **種目選択は2段階**: 部位カテゴリ → 種目一覧（100以上の種目があるため一覧から直接選ばせない）
- **前回コピー機能**: 前回のセット記録をワンタップで入力フィールドにコピーできる
- **重複防止**: 同日・同顧客のセッション作成時に警告を表示する
- **論理削除のみ**: 顧客・トレーナー・セッションはすべて論理削除。物理削除は行わない
- **プライマリオーナー保護**: `is_primary: true` のオーナーは削除不可
- **Vercel Cron**: 毎日0:00 JST に in_progress セッションを自動で completed に変更
- **アプリ名は定数管理**: 後から変更しやすいよう、アプリ名は一箇所で定義する
